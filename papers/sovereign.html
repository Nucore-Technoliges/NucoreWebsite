Sovereign Inference: Cryptographic Attestation for On-Premise Frontier Models

Abstract

Sovereign AI deployments — classified networks, regulated industries, national security environments — require verifiable answers to two questions: which model weights executed, and on which physical hardware. Existing solutions rely on software attestation, which can be spoofed by a compromised hypervisor or operating system. We present a hardware-rooted attestation protocol for Nucore Spartan that signs the active model weights, the firmware revision, and the per-request inference trace, producing an auditable chain of custody verifiable by any third party with the public attestation key. The protocol adds 0.4 ms of overhead per request and has been deployed on classified-network Spartan clusters across three sovereign AI programs since June 2025, with zero false attestations recorded across 847 million attested requests.

1. Introduction

As frontier language models are deployed in sensitive environments — defense intelligence analysis, classified research, regulatory compliance workflows, critical infrastructure — the question of model identity becomes security-critical. A compliance officer must be able to answer: "When this decision was made using AI, exactly which model made it?" A national security analyst must be able to verify: "This analysis was produced by the approved model version, running on approved silicon, and has not been intercepted or tampered with."

Software attestation — having the software stack report its own configuration — is insufficient for high-assurance environments because it can be spoofed by a compromised operating system, hypervisor, or container runtime. What is needed is hardware-rooted attestation: a cryptographic signature produced by the physical silicon that cannot be forged by any software-level compromise.

This paper describes the design, implementation, and production deployment of a hardware-rooted attestation protocol for Spartan that provides this assurance.

2. Threat Model

2.1 Adversary Capabilities

We consider adversaries with the following capabilities:

  • Software compromise: The adversary has root access to the operating system and hypervisor running on the Spartan host. They can modify software, intercept system calls, and inject code into running processes.
  • Weight substitution: The adversary can substitute a different set of model weights (e.g., a backdoored or censored version) for the approved weights at runtime.
  • Response fabrication: The adversary can intercept model outputs and substitute fabricated responses that appear to come from the legitimate model.

We do not consider adversaries with physical access to unseal the Spartan silicon package and probe the hardware directly. Physical attacks on sealed silicon are outside the practical threat model for the environments we target.

2.2 Security Goals

The protocol must provide the following guarantees:

  1. Weight identity: The attestation proves that a specific, approved set of model weights (identified by a cryptographic hash) executed the request.
  2. Hardware identity: The attestation proves that the computation ran on a specific, identified piece of Spartan silicon (identified by a per-device key provisioned at manufacturing time).
  3. Firmware integrity: The attestation proves that the approved firmware revision was active, ruling out firmware-level compromise.
  4. Request binding: The attestation is bound to the specific inference request (input prompt + output tokens), preventing replay attacks with prior attestations.

3. Protocol Design

3.1 Hardware Root of Trust

Each Spartan accelerator is provisioned at manufacturing time with a unique Ed25519 key pair stored in a write-once, read-protected memory region on the silicon die. The private key never leaves the silicon; the public key is registered with Nucore's attestation certificate authority at provisioning and can be queried by any party authorized to verify attestations.

The attestation signing operation runs in a dedicated hardware security module (HSM) subsystem on the Spartan die, physically isolated from the tensor-core compute fabric. The HSM can only be triggered by the firmware — not by software running above the driver layer — ensuring that a software-level adversary cannot generate fake attestations.

3.2 Attestation Payload Structure

For each inference request, the HSM assembles and signs an attestation payload containing:

  • device_id: The silicon serial number (256-bit)
  • firmware_hash: SHA-3-256 hash of the active firmware image
  • weight_hash: SHA-3-256 hash of the loaded model weights, computed at load time and cached in the HSM
  • request_hash: SHA-3-256 hash of the input prompt (after tokenization)
  • response_hash: SHA-3-256 hash of the generated output token sequence
  • timestamp: Hardware clock timestamp with microsecond resolution
  • signature: Ed25519 signature over the concatenation of the above fields

The attestation payload is returned alongside the inference response. The receiving party verifies the signature against the device's registered public key (retrieved from the Nucore attestation CA or a local certificate store in air-gapped deployments).

3.3 Weight Hash Caching and Update

Computing the SHA-3-256 hash of full model weights (approximately 140 GB for Titan-70B in BF16) is expensive if done per request. The HSM caches the weight hash at model load time and invalidates it when any portion of loaded weights changes. The weight hash computation at load time takes approximately 2.8 seconds on Spartan's DMA engine; this is a one-time cost amortized across all subsequent inference requests, contributing less than 0.01 ms to per-request attestation overhead.

4. Performance Impact

The attestation protocol adds 0.4 ms of overhead per request, measured as the additional latency introduced by the HSM signing operation. This overhead is dominated by the Ed25519 signing operation (approximately 0.32 ms) and the SHA-3-256 hash of the request and response (approximately 0.06 ms combined for typical prompt + response lengths).

The 0.4 ms overhead represents 3.3% of the median TTFT (11.8 ms) reported in our concurrent production study. For latency-sensitive deployments that cannot accept this overhead, attestation can be configured for a sampled fraction of requests (default: 100% sampling in classified deployments, 10% in commercial high-throughput deployments) with the full audit trail reconstructable from the sampled attestation log.

5. Deployment Experience

The attestation protocol has been deployed on Spartan clusters in three sovereign AI programs since June 2025. Over this period, we have recorded 847 million attested inference requests with zero false attestations (attestations that verify correctly but correspond to a request where the signed conditions were not actually met).

We have recorded two attestation failures: cases where the attestation signature could not be verified by the receiving party. Both were attributable to clock skew between the Spartan hardware clock and the verifier's expected timestamp range (a configuration issue, not a protocol flaw), and were resolved by synchronizing the acceptable timestamp window in the verification configuration.

6. Conclusion

Hardware-rooted attestation for LLM inference provides a level of assurance not achievable by software-only approaches. By anchoring the attestation signature in silicon-level keys that cannot be accessed by software, our protocol makes it cryptographically infeasible for a software-level adversary to forge an attestation, substitute model weights, or fabricate responses without detection. The 0.4 ms overhead makes the protocol practical for production deployments.
