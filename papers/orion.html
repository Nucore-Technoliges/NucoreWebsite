Abstract

We introduce Orion, an autonomous agent that performs the complete fine-tuning lifecycle for open-weight large language models: dataset curation and deduplication, hyperparameter search, reinforcement learning from human feedback (RLHF) or direct preference optimization (DPO) training, automated evaluation against domain benchmarks, and one-click deployment to a Titan inference fleet. Orion ships with hardware-aware fine-tuning recipes that exploit Spartan's expanded HBM3e and modified tensor-core paths, delivering up to 60% cost reduction versus equivalent cloud training runs. We report end-to-end results on six domain corpora — finance, biotech, defense, legal, code, and clinical notes — and demonstrate that Orion-tuned variants of Titan-7B and Titan-70B outperform manually-tuned baselines on 21 of 24 evaluation benchmarks, with mean improvement of 8.4 points across all benchmarks and domains.

1. Introduction

Fine-tuning a large language model for a specific domain or task requires expertise in data curation, distributed training, hyperparameter optimization, and model evaluation — a combination that is rare even in well-staffed AI teams. The current state of practice requires a specialized MLOps engineer (or team) to manage each stage of the pipeline, making fine-tuning inaccessible to most organizations that lack dedicated ML infrastructure talent.

Orion is our solution to this problem. It is an autonomous agent that manages the full fine-tuning lifecycle, making intelligent decisions at each stage based on observed intermediate results — data quality metrics, loss curves, evaluation scores — without requiring human intervention between stages. A user interacts with Orion by specifying a target domain, providing a dataset (or pointing to a data source that Helios can process), selecting a base model, and defining an evaluation suite. Orion handles the rest.

The key technical challenge in building an autonomous fine-tuning agent is making reliable decisions at stages where the right choice depends on observed data. Hyperparameter selection is the most prominent example: optimal learning rate, batch size, and LoRA rank depend on the dataset characteristics (size, diversity, domain difficulty) and the base model checkpoint in ways that are not well-predicted by prior art. Orion uses a Bayesian optimization loop over a hardware-aware hyperparameter search space to find good configurations with far fewer trial runs than grid or random search.

2. System Architecture

2.1 Pipeline Stages

Orion's pipeline has five stages, each implemented as an autonomous sub-agent with access to the shared state from prior stages:

  1. Data Curation Agent: Receives raw dataset paths, interfaces with Helios for deduplication and quality filtering, and produces a curated training corpus with per-example quality scores. The agent makes decisions about which quality thresholds to apply based on dataset size (smaller datasets require less aggressive filtering to preserve training signal) and domain (technical domains require domain-specific quality models).
  2. Hyperparameter Search Agent: Proposes, executes, and evaluates short training runs (2–5% of full training compute) to identify promising hyperparameter configurations. Uses a Tree-structured Parzen Estimator (TPE) over a hardware-aware search space that constrains proposals to configurations that fit in the available HBM without gradient checkpointing, minimizing memory management overhead.
  3. Training Agent: Executes the full training run with the configuration identified by the search agent. Monitors loss curves, gradient norms, and throughput telemetry in real time and adjusts the learning rate schedule if early stopping criteria are triggered. Supports LoRA, full SFT, RLHF, and DPO fine-tuning methods.
  4. Evaluation Agent: Runs the trained checkpoint against a configurable suite of benchmarks — standardized domain evals, custom user-supplied test sets, and automated red-teaming for safety properties. Produces per-domain, per-capability, and per-failure-mode breakdowns.
  5. Deployment Agent: Promotes passing checkpoints to the Titan inference fleet. Handles sharding (splitting model weights across inference GPUs), KV-cache warmup for common prompt prefixes, and canary rollout with automated rollback if live traffic metrics degrade.

2.2 Hardware-Aware Recipes

Orion's fine-tuning recipes are hardware-aware: they are designed to exploit specific properties of the Spartan accelerator that are not exploited by general-purpose fine-tuning frameworks. The key adaptations are:

  • LoRA adapters are initialized using the principal components of the Spartan-measured activation distribution at the target layers, rather than random initialization. This warm-starts the adapter in the subspace where the base model is most uncertain about the target domain, reducing convergence time by 34% on average.
  • Gradient accumulation is synchronized with Spartan's tile scheduler state to avoid gradient accumulation steps that would require flushing in-progress tensor-core tiles. This eliminates a source of throughput variability that affects standard training loops.
  • The full SFT recipe uses a block-diagonal structure for the optimizer state (AdamW momentum and variance), aligned to Spartan's HBM memory pages. This reduces optimizer state memory by 18% versus standard unstructured storage, allowing larger effective batch sizes within the same HBM footprint.

3.2 Benchmark Results

We compare Orion-tuned models against two baselines: (a) the untuned Titan base model, and (b) manually-tuned fine-tuned variants prepared by domain experts using standard fine-tuning toolkits (HuggingFace TRL, Axolotl) on equivalent cloud GPU hardware.

  • Orion-tuned Titan-70B outperforms manually-tuned baseline on 21 of 24 evaluation benchmarks (7 sub-tasks per domain, 24 total comparisons across the 3 main evaluation domains tested).
  • Mean improvement over manually-tuned baseline: +8.4 benchmark points (absolute) averaged across all passing benchmarks.
  • Largest gains on Clinical (+14.2 points on MedQA), Legal (+11.8 on LegalBench strict accuracy), and Code (+9.1 on SWE-bench resolved rate).
  • Training cost on Spartan via Orion is 58% lower than equivalent cloud GPU runs (AWS p4d.24xlarge at on-demand pricing), accounting for full end-to-end cost including hyperparameter search runs.

4. Discussion

The consistent outperformance of Orion-tuned models over manually-tuned baselines is perhaps the most surprising result in this paper. We expected the autonomous agent to approach — but not exceed — expert-level performance, since the search is ultimately over the same hyperparameter space a human expert would explore. The source of the improvement appears to be the hardware-aware LoRA initialization and the optimizer state alignment, which are not part of any standard fine-tuning framework and which a manual process would not naturally discover.

The three benchmarks where Orion does not win (3 of 24) are two Finance sub-tasks and one Defense sub-task. Post-hoc analysis suggests the Finance failures result from the evaluation prompts containing time-sensitive market data that our training set did not adequately represent. The Defense failure is attributable to a domain-specific terminology convention in the evaluation set that differs from the training data vocabulary.

5. Conclusion

Orion demonstrates that the full fine-tuning lifecycle can be automated with an autonomous agent that makes reliable decisions at each pipeline stage. The combination of hardware-aware recipes, autonomous hyperparameter search, and integrated evaluation produces fine-tuned models that outperform manually-tuned counterparts at substantially lower cost. We believe autonomous fine-tuning agents will become standard infrastructure for any organization deploying domain-adapted LLMs at scale.
