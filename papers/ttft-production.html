Sub-12ms Time-to-First-Token at 1,024-Sequence Concurrency: A Production Study

Abstract

We present a 90-day production trace from three Nucore Spartan deployments serving agentic AI workloads at 1,024-sequence concurrency. We characterize the full latency distribution under realistic traffic conditions, isolate the contribution of KV-cache persistence to tail latency, and demonstrate that hardware-aware batch construction eliminates 78% of the cold-start penalty typical of cloud-hosted inference. Our median time-to-first-token (TTFT) across the 90-day trace is 11.8 ms, with p99 at 38 ms. We open-source a reference scheduler implementation and a synthetic agent-traffic benchmark that reproduces the traffic patterns observed in production.

1. Introduction

Time-to-first-token (TTFT) is the latency from when an inference request is received to when the first output token is generated. For agentic AI workloads — where models are called repeatedly within a reasoning loop, often with overlapping context windows from prior turns — TTFT is the primary latency metric that determines the responsiveness of the overall system. A reasoning loop with 100 model calls at 12 ms TTFT takes 1.2 seconds of model latency; at 120 ms TTFT (typical for cloud-hosted inference at moderate concurrency), the same loop takes 12 seconds.

Achieving sub-12ms TTFT at production-level concurrency (hundreds to thousands of simultaneous sessions) requires solving three problems simultaneously: scheduling inference requests efficiently so that GPU compute is never idle while queued requests wait, keeping KV-cache data in fast memory so that attention computation over prior context does not stall on HBM reads, and constructing batches in a way that avoids the cold-start penalty caused by a newly-arrived request having no pre-computed key-value state.

This paper reports our experience solving these problems in production on the Spartan platform, with empirical measurements from three customer deployments spanning 90 days of continuous operation.

2. Production Environment

2.1 Deployment Configurations

We report on three deployments, which we anonymize as D1, D2, and D3:

  • D1 (Financial services, 128 GPUs): Serving a multi-agent portfolio analysis system. Mean session length: 4,200 tokens of context. Peak concurrency: 1,024 sessions. 90-day trace: 847 million requests.
  • D2 (Defense research, 64 GPUs, air-gapped): Serving a document analysis and synthesis agent. Mean session length: 18,400 tokens. Peak concurrency: 312 sessions. 90-day trace: 142 million requests.
  • D3 (Enterprise SaaS, 256 GPUs): Multi-tenant agent platform serving 14 different corporate customers. Mean session length: 2,800 tokens. Peak concurrency: 3,200 sessions. 90-day trace: 2.1 billion requests.

All deployments run Titan-70B (BF16 weights, 192 GB HBM3e per accelerator) with tensor parallelism across 8 GPUs per model instance. The scheduler is our reference implementation described in Section 4.

3. Latency Analysis

3.1 Overall TTFT Distribution

Across all three deployments, median TTFT is 11.8 ms (p50), p90 is 21.4 ms, p99 is 38.2 ms, and p99.9 is 112 ms. The p99.9 tail is dominated by requests that arrive during scheduler preemption events, which occur when the KV cache approaches capacity and lower-priority sessions must have their cache state evicted to accommodate higher-priority requests.

The 38 ms p99 represents a 4.1× improvement over our internal baseline measurement taken on an equivalent H100 deployment (157 ms p99 at the same concurrency level), and a 6.3× improvement over the best publicly-reported figure we are aware of for cloud-hosted inference at 1,024-way concurrency.

3.2 Cold-Start Penalty

A "cold-start" request is one where the model has no pre-computed KV-cache state for the session — either a new session or a session whose cache was evicted. Cold-start requests must compute attention over the full context prefix before generating the first output token, which at 70B scale and 4,096-token contexts takes approximately 84 ms on H100 hardware. On Spartan with persistent HBM3e cache, cold-start cost is reduced to approximately 19 ms, a 4.4× improvement attributable to (a) the expanded HBM pool allowing more session caches to remain resident and (b) higher HBM bandwidth enabling faster context prefill when a cold start does occur.

The 78% cold-start reduction reported in the abstract refers to the reduction in the fraction of requests that experience a cold start (from 22% on H100 to 4.8% on Spartan), driven by the larger in-HBM cache footprint that keeps more sessions warm.

3.3 Concurrency Scaling

We measure TTFT as a function of concurrency from 64 to 1,536 simultaneous sessions on D3. On H100 hardware (extrapolated from a matched subset of the deployment), TTFT begins to degrade sharply above 512 sessions as the KV cache fills and eviction becomes frequent. On Spartan, TTFT remains below 15 ms median up to 1,024 sessions, with noticeable degradation only above 1,280 sessions. The crossover point — above which Spartan's larger cache pool provides material advantage — is approximately 640 sessions.

4. Hardware-Aware Batch Construction

The reference scheduler we developed for Spartan deployments uses hardware-aware batch construction, which we define as the practice of constructing inference batches with explicit knowledge of KV-cache residency state and GPU memory pressure. The key ideas are:

  • Cache-residency-aware prioritization: Requests for sessions whose KV cache is currently in HBM are prioritized over requests for cold sessions, as they can be scheduled immediately without prefill cost. This has the counter-intuitive effect of reducing mean TTFT by accepting slightly higher worst-case latency for cold requests.
  • Prefill-decode disaggregation: Prefill (computing KV state for the input prefix) and decode (generating output tokens one at a time) are physically disaggregated across GPU subsets. Prefill is GPU-compute-heavy; decode is memory-bandwidth-heavy. Running them on separate GPU subsets prevents decode throughput from being limited by co-located prefill compute.
  • Speculative cache pre-warming: When a session submits a request, the scheduler begins pre-computing KV state for the predicted next turn's context (using the most likely continuation tokens from the current output distribution). On average, pre-warming reduces next-turn TTFT by 38% for agentic sessions with predictable turn structure.

6. Conclusion

We have demonstrated sub-12ms median TTFT at 1,024-sequence concurrency on production Spartan deployments serving agentic AI workloads. The result is achieved through the combination of Spartan's hardware properties (192 GB HBM3e per accelerator, persistent cache architecture) and the hardware-aware scheduler that exploits those properties. The reference scheduler and benchmark harness are open-sourced to allow the community to evaluate and extend these techniques on other hardware configurations.
